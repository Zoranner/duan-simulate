## 问题描述

在规划探测域的 `compute` 逻辑时，我发现一个根本性的设计困惑：

`DomainContext` 的 `own_entities` 字段（见 `domain.md`）只包含"当前域管辖的实体 ID 集合"。但探测是一个**跨实体的交叉判断**：观察者（属于探测域）需要遍历潜在目标，而目标并不一定也属于探测域。

例如：
- 驱逐舰有雷达，归属探测域（是探测者）
- 潜艇没有雷达，不归属探测域，但可以**被**探测到（是探测目标）

如果探测域的 `compute` 中只能访问 `ctx.own_entities`（即归属探测域的实体），那么潜艇这种"只能被探测，不能主动探测"的实体永远不会出现在探测目标列表中。

这造成了一个概念上的困境：

1. **让所有可被探测的实体都加入探测域**？——语义上不对，"被探测"不是一种能力，而是一种属性。
2. **从 `ctx.entities` 获取全量实体**？——`domain.md` 的"域上下文"表格显示 `entities` 字段是可变访问，但没有说明是否支持按条件查询所有活跃实体（只读遍历）。
3. **通过空间域的服务接口做范围查询**？——空间域的 `范围内实体查询` 服务返回的是什么范围的实体？是全局的还是空间域管辖的？

文档没有解释这种跨实体交叉计算场景下应该使用哪种模式。

## 影响程度

- [x] 中等（影响开发效率或理解，有变通方式）

## 复现场景

设计探测域时，需要对每个"有雷达的实体"遍历所有"可被探测的实体"（通常是全部活跃实体，或某个空间范围内的实体）：

```rust
fn compute(&mut self, ctx: &mut DomainContext) {
    let observers: Vec<EntityId> = ctx.own_entity_ids().collect(); // 有探测能力的实体

    for observer_id in observers {
        // 目标应该是"所有活跃实体"，但如何获取？
        // ctx.entities.all_active() ?  —— 不确定此方法是否存在
        // ctx.entities.iter() ?        —— 不确定是否遍历全量还是仅本域
        for target_id in ??? {
            if target_id == observer_id { continue; }
            // 计算能否探测到...
        }
    }
}
```

## 建议方案

**需架构讨论**：

这涉及框架对 `ctx.entities` 接口的设计决策。以下是两种可能方向：

**方向一：ctx.entities 支持全量只读遍历**

`ctx.entities` 提供一个 `iter_active()` 或 `all_active_ids()` 方法，返回所有活跃实体的只读迭代器，不限于当前域的管辖范围。

优点：直接、灵活。缺点：域可以读取到并非自己管辖的实体，权威边界稍有模糊（但读取不修改，实际上是安全的）。

**方向二：通过空间域服务统一处理目标枚举**

空间域维护全局空间索引，提供"以某位置为中心、某半径范围内的所有实体"服务。探测域通过依赖空间域的这个服务来枚举候选目标，而不是自己遍历全局。

优点：职责清晰，范围查询高效。缺点：要求空间域的服务接口确实是全局范围的（文档未明确）。

**短期可改进**：

无论采用哪种方向，`domain.md` 的"域上下文"表格和 `custom-domain.md` 的参考实现中，都应该明确回答：

1. `ctx.entities` 支持哪些遍历方式？只读和可变各有哪些？
2. 探测类"双循环"场景（观察者 × 目标）的推荐实现模式是什么？
3. 空间域的范围查询服务是针对全局还是本域管辖的实体？

## 维护者评估

**结论**：采纳。问题成立，但不是架构缺陷——框架实现中全量遍历能力已存在，根本原因是文档描述不完整造成了概念困惑。同时需要在文档中明确表达"全量只读遍历合法"这一设计立场。

**分析**：

经核查 `src/entity.rs`，`EntityStore` 提供：
- `iter()` —— 迭代全部实体（含任意生命周期状态）
- `active_entities()` —— 只返回 `Lifecycle::Active` 的实体
- `by_type(entity_type)` —— 按类型过滤

`ctx.entities` 是完整的 `&mut EntityStore`，因此域在 `compute` 中可以直接调用 `ctx.entities.active_entities()` 获取全量活跃实体。Reporter 的困惑来自文档而非实现。

**架构设计立场**：

全量只读遍历是**合法的**，不违反权威域架构。

权威边界的核心约束是**写入**：域只能修改自己管辖的实体组件状态，不能越权修改其他域管辖的实体。但**读取**任意实体的数据不破坏任何域的权威——这与 `ctx.registry` 允许只读查询任意已注册域的服务是同一逻辑。

因此：
- `ctx.entities.active_entities()` 遍历全量实体 → **合法，且已支持**
- 修改非管辖实体的组件状态 → 越权，应避免（框架不强制，由开发者自律）

Reporter 提出的方向二（通过空间域服务返回范围内实体）是探测场景下**更推荐**的写法，因为它避免了 O(n) 全量遍历、利用空间索引加速。但这是性能建议，不是强制约束。文档应明确两种方式都是合法的，并说明适用场景。

文档中 `domain.md` 将 `entities` 字段描述为"读写组件数据"，暗示了只操作管辖实体，但从未明确说明"只读遍历全量实体"是否允许，造成了 reporter 的合理困惑。

**行动计划**：

1. 在 `concepts/domain.md` 的"域上下文"表格中，补充对 `entities` 字段访问能力的完整描述：支持全量只读遍历（`active_entities()`），写入权限仅限于自身管辖实体
2. 在 `guides/custom-domain.md` 的探测域参考实现中，补充"遍历目标实体"的说明：两种方式（全量遍历 vs 通过空间域服务），注明各自适用场景
