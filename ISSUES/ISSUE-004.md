## 每处修改的摘要和理由

### overview.md — 初始化顺序删除"注册事件处理器"

**原文**：

```
注册域（含执行顺序解析）
注册事件处理器
加载场景（创建初始实体）
```

**修改后**：

```
注册域（含执行顺序解析）
加载场景（创建初始实体）
```

补充说明：事件处理器不是注册式的——框架通过 `step_with` 的闭包参数接受自定义事件处理逻辑。

**理由**：框架 `World` API 中不存在"注册事件处理器"的接口。自定义事件处理通过 `step_with(dt, |event, world| {...})` 传入，是调用时传参而非初始化时注册。原描述会让开发者以为存在类似 `world.register_handler(...)` 的 API。

### overview.md — 计算阶段数据流图补充直接写入路径

**原文**：

```
域读取实体数据 → 域执行计算 → 域发出事件 → 事件进入事件通道
```

**修改后**：增加"直接写入自身管辖实体的组件状态"分支，并澄清事件处理阶段中 `step_with` 闭包和框架内置处理的区别。

**理由**：原图遗漏了域直接修改组件状态这条核心路径，与 ISSUE-003 修正后的设计哲学（域在计算阶段有完整写入权）不一致。仅展示事件路径的数据流图会误导读者以为"所有状态修改都通过事件完成"。

### lifecycle.md — Initializing 状态补充"瞬间性"说明

**原文**：未说明该状态的持续时间特征。

**修改后**：在状态描述中补充"这是 `World::spawn` 内部的同步瞬间状态——实体从构造到活跃在同一次 `spawn` 调用中完成，不会跨帧停留在此状态"。

**理由**：原文描述给人印象是 `Initializing` 为一个可持续存在的稳态，开发者可能误以为在某些情况下需要等待其完成。实际实现中 `spawn()` 是同步操作，在一次函数调用中完成 `Initializing → Active` 的完整转移，这个状态对外部（计算阶段、仿真循环）是不可见的。澄清这一点有助于开发者正确理解实体创建后立即参与计算的行为。

## 是否涉及概念定义或架构边界调整

**overview.md 修改**：涉及初始化顺序和数据流架构描述调整，属于**重要变更**——纠正了初始化顺序中的错误描述（删除不存在的注册步骤），补充了计算阶段的核心数据流路径。

**lifecycle.md 修改**：是对现有概念的精度补充，不涉及概念边界调整，属于轻微变更。
