# 决策记录

本文档记录 DUAN 仿真体系在设计过程中识别的问题、已做出的决策及尚未解决的权衡点。

---

## 已解决的问题

### 生命周期：Initializing 状态的语义

**问题**：初始化中（Initializing）状态下实体是否参与计算？文档早期存在矛盾——一处说"可能参与（取决于域的实现）"，另一处说"尚未完成域归属校验"。

**决策**：Initializing 状态的实体**不参与任何域的计算**。此时实体尚未加入任何域的实体列表，域计算时不会处理它。校验完成后，无论哪些域通过，实体立即进入 Active 状态。

---

### 准入失败后实体的状态

**问题**：如果部分域拒绝了实体，实体处于什么生命周期状态？是否进入 Active？

**决策**：准入校验失败**不阻止实体进入 Active 状态**。实体进入 Active 后，只是不在拒绝它的域的实体列表中。框架应记录准入失败的域及原因，供用户排查。

---

### 事件处理阶段产生新事件的规则

**问题**：simulation-loop 和 event 两份文档对"处理器能否产生新事件"的描述相互矛盾。

**决策**：事件处理器可以产生新事件，但**新事件在下一帧的事件处理阶段处理**，不在本帧处理。这保证单帧内处理是有界的，不会因事件链式触发导致无限循环。

---

### 命名：域驱动设计（DDD）

**问题**：使用 "Domain-Driven Design" 这个已有的软件工程术语，但 DUAN 的"域"与 DDD 的"领域"含义不同，会引起熟悉 DDD 的读者的强烈误解。

**决策**：改用 "Domain-Authoritative Architecture（权威域架构）"，突出 DUAN 核心的"权威"语义，与 ECS System 的区别也更清晰。

---

### 销毁中状态与前端耦合

**问题**：Destroying 状态的设计理由被描述为"执行销毁动画"，将具体的应用层行为写入了框架层概念。

**决策**：框架层只保证"Destroying 状态下实体不参与计算，提供过渡期"。过渡期的具体用途（动画、通知、清理等）由应用层自行决定。

---

### step_with 闭包签名与 CustomEvent 的 object lifetime

**问题**：框架在 `impl dyn CustomEvent` 上提供的 `downcast<T>()` 便捷方法，在 `step_with` 闭包内无法编译（E0521）。根因分析经历了两个阶段：

- **第一阶段分析（ISSUE-002）**：认为问题在于 `downcast` 方法签名缺少显式生命周期参数，修复为 `pub fn downcast<'a, T: 'static>(&'a self) -> Option<&'a T>`。修复后编译仍然失败。
- **第二阶段分析（ISSUE-005）**：发现真正根因在 `step_with` 的闭包约束。`impl dyn CustomEvent` 在 Rust 中隐式等价于 `impl (dyn CustomEvent + 'static)`，因此 `downcast` 要求 `self` 的 object lifetime 满足 `'static`。而 `step_with` 原有约束 `F: FnMut(&dyn CustomEvent, &mut Self)` 并未明确 object lifetime，导致 Rust 将闭包参数推断为短生命周期引用，无法调用 `impl (dyn CustomEvent + 'static)` 上的方法。

**决策**：将 `step_with`、`do_step`、`process_events` 的闭包约束由 `F: FnMut(&dyn CustomEvent, &mut Self)` 改为 `F: FnMut(&(dyn CustomEvent + 'static), &mut Self)`，明确要求传入的事件对象满足 `'static` object lifetime。这与框架内部实际传入的类型一致（`DomainEvent::custom<E: CustomEvent + 'static>` 存储 `'static` object），是语义明确化而非行为变更。

**注意**：使用 HRTB（`for<'e> FnMut(&'e (dyn CustomEvent + 'e), ...)`）无法解决此问题，因为它只改变生命周期的展开方式，闭包参数的 object lifetime 仍不确定，`impl (dyn CustomEvent + 'static)` 上的方法仍不可调用。

## 待观察的设计点

以下问题没有最优解，需要在实践中验证：

### 域粒度的选择

域的粒度太粗（一个域负责太多事情）或太细（域过多，协调复杂）都有问题。目前文档给出了建议，但没有量化标准。

**当前策略**：建议每个域对应一个相对独立的"领域"，在实践中根据依赖关系的复杂度决定是否需要拆分或合并。

---

### 组件粒度的选择

与域粒度类似，组件应该多粗？粒度太细导致同时出现的组件太多，粒度太粗导致复用困难。

**当前策略**：每个组件描述实体的一个逻辑方面。如果两个字段总是同时出现且总是一起被读取，就放在同一个组件中。

---

### 定时器的创建时机

定时器通常在事件处理器中创建。目前没有明确禁止在域的计算阶段创建定时器，但计算阶段创建定时器会在同帧的定时器检查阶段之后才有效（因为定时器检查先于事件处理）。

**当前策略**：推荐在事件处理器中创建定时器，在计算阶段创建定时器的效果和时序留待实现时明确。

---

### 并行化域计算

当前设计是串行执行各域计算。文档中提到"便于并行化（未来扩展）"，但没有详细设计。

如果引入并行化，需要解决：
- 同一帧内域之间的服务查询（并行时被查询的域可能还没有运行本帧计算）
- 内部状态的线程安全

**当前策略**：暂不引入并行化，保持串行确定性。如果性能成为瓶颈，再专项设计并行方案。
