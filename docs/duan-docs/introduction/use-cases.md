# 适用场景

本文档描述 DUAN 仿真体系的典型应用场景，帮助判断是否适合使用 DUAN。

## 适合的场景

### 多领域交叉仿真

当仿真涉及多个相对独立的领域，且这些领域之间需要交互时，DUAN 的域驱动设计非常适用。

**例子**：

- 军事推演：需要同时模拟运动、探测、通信、交战等多个领域
- 物理仿真：需要同时模拟流体、刚体、热传导等多个物理过程
- 社会模拟：需要同时模拟经济、政治、文化等多个社会子系统

**为什么适合**：

- 每个领域可以独立建模和测试
- 领域之间的交互通过事件系统清晰地表达
- 新增领域不影响现有领域

### 需要灵活扩展的仿真

当仿真需求会随时间演进，需要不断添加新的能力时，DUAN 的可扩展性非常有价值。

**例子**：

- 游戏开发：初期可能只需要移动和碰撞，后期可能需要添加技能系统、天气系统等
- 科研仿真：可能需要不断添加新的模型和算法
- 教学演示：可能需要根据课程进度逐步引入新的概念

**为什么适合**：

- 新增域不需要修改现有代码
- 域的依赖关系自动管理
- 可以逐步构建仿真系统

### 需要替换算法的仿真

当同一仿真逻辑可能有多种实现（简化版、真实版、不同精度版），且需要灵活切换时，DUAN 的插件化设计很有用。

**例子**：

- 概念验证阶段使用简化算法，后期切换到精确算法
- 不同场景使用不同精度的物理计算
- 调试时使用确定性算法，发布时使用随机化算法

**为什么适合**：

- 域的算法实现是可替换的
- 切换算法不影响实体和场景配置
- 算法参数可以动态调整

### 需要可追溯性的仿真

当仿真的结果需要能够追溯和分析时，DUAN 的事件驱动设计提供了良好的支持。

**例子**：

- 军事推演需要事后复盘
- 科学实验需要记录过程
- 游戏需要回放功能

**为什么适合**：

- 所有计算结果都通过事件表达
- 事件可以完整记录
- 可以基于事件日志进行回放和分析

## 不太适合的场景

### 简单的单一领域仿真

如果仿真只涉及一个领域，且不需要扩展，使用 DUAN 可能过于复杂。

**建议**：直接实现，不需要框架。

### 极致性能要求的场景

如果对性能有极致要求，无法接受事件系统的开销，DUAN 可能不是最佳选择。

**建议**：考虑更紧密耦合的设计，或者专门的引擎。

### 需要网络同步的场景

DUAN 是单机引擎，不处理网络同步。

**建议**：在 DUAN 之上构建网络层，或者选择支持网络的引擎。

### 需要可视化支持的场景

DUAN 不提供可视化功能。

**建议**：将 DUAN 作为仿真后端，配合专门的可视化前端使用。

## 典型用户画像

### 仿真系统开发者

需要在短时间内构建一个可扩展的仿真系统，DUAN 提供了良好的骨架。

### 游戏开发者

需要构建复杂的游戏逻辑，且预期会不断扩展，DUAN 的域驱动设计可以帮助组织代码。

### 科研人员

需要进行可重复、可分析的仿真实验，DUAN 的事件驱动设计便于记录和分析。

### 教育工作者

需要构建教学用的仿真系统，DUAN 的清晰架构便于学生理解仿真原理。

## 成功案例

> 注：以下为假设性案例，用于说明 DUAN 的适用场景。

### 案例一：海战推演系统

**背景**：需要模拟多艘舰船在海域中的运动、探测、交战过程。

**使用方式**：

- 定义"空间域"处理位置和距离
- 定义"探测域"处理雷达探测
- 定义"战斗域"处理武器交战
- 定义"阵营域"处理敌我关系

**收益**：各领域的逻辑独立开发和测试，通过事件系统协调。

### 案例二：多物理场仿真

**背景**：需要模拟流体、热传导、结构应力的耦合过程。

**使用方式**：

- 定义"流体域"处理流体动力学
- 定义"热传导域"处理温度分布
- 定义"结构域"处理应力应变
- 各域通过事件传递边界条件

**收益**：每个物理场可以独立建模，耦合关系清晰表达。

### 案例三：游戏技能系统

**背景**：需要为游戏角色实现复杂的技能交互系统。

**使用方式**：

- 定义"移动域"处理角色移动
- 定义"战斗域"处理伤害计算
- 定义"状态域"处理 buff/debuff
- 定义"触发域"处理条件触发

**收益**：技能系统可以独立于核心玩法开发，易于扩展新技能。
