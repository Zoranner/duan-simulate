# 设计目标

本文档描述 DUAN 仿真体系的设计目标，帮助理解框架的设计决策和使用方式。

## 核心目标

### 通用性

**目标**：框架核心不关心具体有哪些域，域类型在运行时定义。

**意义**：

- 框架可以应用于完全不同的仿真领域
- 用户不需要 fork 或修改框架来适应新场景
- 学习一次框架，应用到多种场景

**体现**：

- 域使用字符串标识，不使用枚举
- 框架不预设任何具体的域类型
- 所有域通过统一的接口注册和使用

### 可扩展性

**目标**：可随时添加新域类型，不改核心代码。

**意义**：

- 仿真需求可以逐步演进
- 新功能可以通过新增域来实现
- 不同项目可以有不同的域组合

**体现**：

- 新增域只需实现约定接口并注册
- 现有域不需要修改就能与新域协作
- 域的依赖关系通过声明自动解析

### 多域归属

**目标**：实体可同时属于多个域，实体自声明归属。

**意义**：

- 实体可以具有多种能力
- 能力的组合由实体自己决定
- 域的接纳有明确的规则

**体现**：

- 实体创建时声明要加入哪些域
- 每个域有准入校验机制
- 一个实体可以同时被多个域管理

### 插件化算法

**目标**：同一域可有简化版/真实版算法实现。

**意义**：

- 开发阶段可以使用简化算法快速迭代
- 发布阶段可以切换到真实算法
- 不同精度需求的场景可以使用不同实现

**体现**：

- 域的算法实现是可替换的
- 切换算法不需要修改实体或场景配置
- 算法参数可以动态调整

### 独立包

**目标**：仿真体系作为独立通用包，可跨项目复用。

**意义**：

- 多个项目可以共享同一套仿真体系
- 仿真体系可以独立演进和优化
- 降低项目的维护成本

**体现**：

- 核心接口稳定，版本化管理
- 与具体应用解耦
- 清晰的分层设计

### 高性能

**目标**：支持大规模实体，组件查询高效。

**意义**：

- 可以处理包含大量实体的场景
- 实时仿真能够保持流畅
- 资源消耗可控

**体现**：

- 组件查询时间复杂度为常数级
- 域可以建立自己的索引结构
- 事件处理采用批量方式

## 设计权衡

### 灵活性 vs 简单性

DUAN 选择了灵活性。这意味着：

- 框架不提供开箱即用的仿真行为
- 用户需要自己定义和实现具体的仿真逻辑
- 初次使用的学习成本较高

这是有意为之的设计决策：框架的价值在于提供机制，而不是策略。一个灵活的框架长期来看比一个充满内置假设的框架更有价值。

### 解耦 vs 性能

DUAN 选择了解耦。这意味着：

- 域之间通过事件通信，有额外的开销
- 计算阶段和更新阶段分离，增加了一步
- 相比紧密耦合的设计，理论性能略低

但是，这种设计带来的好处是：

- 调试更容易（事件可追踪）
- 回放更容易（事件可记录）
- 理解更容易（职责清晰）

### 抽象 vs 具体性

DUAN 选择了抽象。这意味着：

- 框架不预设任何具体的仿真概念
- 用户需要自己定义"舰船"、"雷达"、"命中"等概念
- 文档中的示例只是参考，不是标准

这是为了确保框架的通用性。用户应该根据自己领域的概念来建模。

## 非目标

以下不是 DUAN 的设计目标：

- **可视化**：DUAN 是仿真体系，不提供可视化功能
- **网络同步**：DUAN 是单机引擎，不处理网络同步
- **物理引擎**：DUAN 不是物理引擎，物理计算需要用户实现
- **AI 系统**：DUAN 不提供 AI，实体的行为逻辑需要用户实现
