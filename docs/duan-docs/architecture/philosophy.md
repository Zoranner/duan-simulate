# 设计哲学

本文档阐述 DUAN 仿真体系的核心设计哲学——权威域架构。

## 权威域架构的核心

### 域是权威

这是 DUAN 最核心的设计原则：

> A 能否看见 B、能否命中 B，由对应域说了算。不是实体自己判断，而是由该领域的权威域来裁决。

**为什么这样设计？**

在传统设计中，"能否看见"这类判断可能分散在多个地方：

- 实体 A 的探测方法中
- 实体 B 的可见性属性中
- 某个全局的可见性判断函数中

这导致：

- 逻辑分散，难以维护
- 不同地方的判断可能不一致
- 修改影响范围难以预测

在 DUAN 中，这类判断**统一由对应的域来裁决**：

- "能否看见"由探测域裁决
- "能否命中"由战斗域裁决
- "是敌是友"由阵营域裁决

### 域运行时定义

> 核心架构不枚举域类型，运行时注册。域标识使用字符串。

**为什么这样设计？**

如果域类型是框架预设的（比如用枚举定义），那么：

- 添加新域需要修改框架代码
- 不同用户需要的域类型不同
- 框架会变得臃肿

通过运行时注册：

- 用户可以根据需要定义任意域
- 框架保持精简
- 扩展不影响框架

### 实体自声明域归属

> 实体声明自己要加入哪些域。域在接纳实体时会进行校验。

**为什么这样设计？**

如果域归属由外部指定（如配置文件或管理器），那么：

- 新增域类型时需要修改配置逻辑
- 域的准入规则难以表达

通过实体自声明：

- 实体知道自己需要什么能力
- 域可以有自己的准入标准
- 双方达成一致才能建立关系

### 域按依赖顺序执行

> 各域声明依赖关系，框架根据依赖关系进行拓扑排序，确保被依赖的域先执行。

**为什么这样设计？**

域之间存在依赖关系：

- 探测域需要知道距离（依赖空间域）
- 战斗域需要知道敌我关系（依赖阵营域）

通过显式声明依赖：

- 执行顺序自动确定
- 新增域不影响现有依赖关系
- 循环依赖可以被检测

### 事件驱动传播

> 域的计算结果通过事件系统传播。事件处理阶段统一更新实体状态。

**为什么这样设计？**

如果域直接修改实体状态：

- 执行顺序会影响结果（A 修改了状态，B 读到的是修改后的值）
- 难以追溯状态变化的原因
- 回放和调试困难

通过事件驱动：

- 计算阶段只读，状态一致
- 所有变化都有事件记录
- 便于追踪和回放

## 设计对比

### 与传统面向对象设计的对比

**传统设计**：

- 实体包含行为（方法）
- "舰船探测"是舰船类的方法
- 逻辑分散在各个类中

**DUAN 设计**：

- 实体只包含数据（组件）
- 探测逻辑在探测域中
- 逻辑集中在对应的域

### 与 ECS 架构的对比

**ECS 设计**：

- System 是计算单元
- System 之间通常平行执行，没有明确的"权威"概念
- 同一类判断可能在多个 System 中各自处理，逻辑分散

**DUAN 设计**：

- 域是**权威计算单元**，是其领域的唯一裁决者
- 域之间有明确的依赖关系，执行顺序自动确定
- 任何关于某个领域的判断，只在一个地方发生

这是 DUAN 的核心差异：域不仅仅是"处理这一类数据的 System"，而是"对这个领域的所有问题拥有最终裁决权的权威"。

## 设计收益

### 清晰的职责划分

每个领域有明确的负责人，不会出现"这段逻辑应该放在哪里"的困惑。

### 可测试性

域可以独立测试，不需要搭建完整的仿真环境。

### 可扩展性

新增领域不需要修改现有代码，只需注册新域并声明依赖。

### 可追溯性

所有状态变化都有事件记录，便于调试和分析。

### 可替换性

域的实现可以替换，便于支持不同精度或算法。

## 设计权衡

### 灵活性 vs 简单性

DUAN 选择了灵活性。框架不提供开箱即用的仿真行为，用户需要自己定义和实现具体的仿真逻辑，初次使用的学习成本较高。这是有意为之的：框架的价值在于提供机制，而不是策略。一个灵活的框架长期来看比一个充满内置假设的框架更有价值。

### 解耦 vs 性能

DUAN 选择了解耦。域之间通过事件通信，有额外的开销；计算阶段和更新阶段分离，增加了一步。但这带来的好处是：调试更容易（事件可追踪）、回放更容易（事件可记录）、理解更容易（职责清晰）。

### 抽象 vs 具体性

DUAN 选择了抽象。框架不预设任何具体的仿真概念，用户需要自己定义"舰船"、"雷达"、"命中"等概念，文档中的示例只是参考，不是标准。这是为了确保框架的通用性——用户应该根据自己领域的概念来建模。
