# 仿真循环

仿真循环是每帧执行的流程。本文档描述仿真循环的阶段和执行方式。

## 仿真循环概述

仿真循环是仿真的心跳，每帧执行一次。每一帧，仿真世界会：

1. 推进时间
2. 执行域计算
3. 检查定时器
4. 处理事件
5. 清理销毁实体

这些阶段按顺序执行，保证确定性和一致性。

## 阶段详解

### 阶段一：时间推进

根据真实时间流逝和时间比例，推进仿真时间。

**输入**：真实帧时间（自上一帧以来经过的真实时间）

**输出**：仿真帧时间（用于后续计算）

**处理**：

- 如果仿真暂停，帧时间为零
- 如果仿真运行，帧时间 = 真实帧时间 × 时间比例
- 更新仿真时钟

### 阶段二：域计算

按拓扑顺序执行各域的计算。

**执行顺序**：

- 由域注册表根据依赖关系计算
- 被依赖的域先执行
- 依赖者后执行

**每个域的操作**：

1. 获取域上下文
2. 调用域的计算接口
3. 域直接修改自身管辖实体的组件状态（域是权威，直接写入）
4. 域查询其他域的服务接口（可选）
5. 域向事件通道追加事件

**关于实体状态的写入权**：

域是其管辖实体的权威，在计算阶段可以**直接修改**所管辖实体的组件状态。这是"域是权威"原则的直接体现——运动域直接积分位置，碰撞域直接修正速度，无需绕道事件系统。

事件用于**跨边界通信**：通知外部观察者、触发生命周期变化、传递跨域信息。

**帧内时序**：

```
域计算阶段（各域 compute 按依赖顺序逐个执行）
│
├── 运动域 compute  →  直接修改 Position / Velocity
├── 碰撞域 compute  →  直接修改 Position / Velocity
│                   └─ emit(GroundCollisionEvent)  ┐
│                                                  │ 事件在此积压
└── （所有域计算完毕）                               │
                                                   ↓
事件处理阶段（统一处理本帧积累的所有事件）
│
├── 用户事件处理器（自定义事件）
└── 框架内置处理（生命周期事件）
```

这一设计保证：某域本帧产生的事件不会影响同帧其他域的计算，仿真行为确定且可复现。

### 阶段三：定时器检查

检查所有到期的定时器，产生定时器事件。

**处理**：

1. 获取当前仿真时间
2. 检查所有定时器
3. 收集到期的定时器
4. 为每个到期的定时器产生事件
5. 如果是周期定时器，重新调度

定时器触发时产生定时器事件，追加到事件通道，在本帧的事件处理阶段统一处理。事件类型由用户在创建定时器时指定。

### 阶段四：事件处理

按顺序处理事件通道中的所有事件。

**处理方式**：

- 串行处理
- 根据事件类型分发到对应的处理器
- 处理器可以修改实体状态

**事件处理的内容**：

- 更新实体状态
- 创建新实体
- 销毁实体
- 触发其他操作

**约束**：

- 此阶段可以修改实体状态
- 事件处理器可以产生新事件，但**新事件在下一帧的事件处理阶段才会被处理**，不在本帧处理

### 阶段五：清理

清理已销毁的实体。实体在进入销毁中状态时已从所有域完全脱离，定时器也已全部取消，清理阶段只需从实体存储中移除。

**处理**：

1. 收集所有已销毁状态的实体
2. 从实体存储中移除这些实体
3. 更新调试信息（如果启用）

## 阶段间的约束

### 计算阶段的写入边界

计算阶段（阶段二）中，域可以修改**自身管辖实体**的组件状态，但不能直接干预其他域的内部状态，也不能发起生命周期操作（创建/销毁实体）。这确保：

- 域之间通过有序的执行依赖协作，而不是相互直接修改
- 生命周期操作集中在事件处理阶段，生命周期变更有迹可查
- 未来并行化时，依赖关系图即为安全的并发边界

### 事件处理阶段的职责

事件处理阶段（阶段四）处理跨边界操作：

- 用户自定义事件回调（如记录碰撞统计）
- 生命周期事件（实体销毁状态转换）
- 定时器触发的延迟操作

### 定时器检查在事件处理之前

定时器事件需要参与事件处理，因此定时器检查在事件处理之前。

## 循环的调用

仿真循环由外部调用驱动。调用方负责：

- 计算真实帧时间
- 调用仿真步骤
- 控制调用频率

典型用法：

- 固定帧率：每固定时间间隔调用一次
- 可变帧率：根据真实时间流逝调用
- 单步执行：手动调用单步

## 确定性

### 什么是确定性

确定性是指：相同的输入和初始条件，产生相同的输出和最终状态。

### 为什么重要

- 可复现：便于调试和测试
- 可回放：可以重现仿真过程
- 可验证：可以验证仿真结果

### 如何保证

- 域的执行顺序固定
- 事件的处理顺序固定
- 不依赖不确定的因素（如随机数需要使用固定种子）
