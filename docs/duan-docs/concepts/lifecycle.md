# 生命周期

生命周期描述实体从创建到销毁的状态变迁过程。本文档描述生命周期的概念、状态和处理流程。

## 什么是生命周期

生命周期是实体存在期间的状态序列。每个实体都有一个生命周期状态，表示其当前所处的阶段。

**为什么需要生命周期**：

- 区分活跃实体和非活跃实体
- 支持销毁前的过渡阶段（如播放动画）
- 便于资源管理和清理

## 生命周期状态

### 已创建（Spawned）

实体刚创建，等待初始化。

**特点**：

- 实体已分配标识
- 组件已初始化
- 尚未完成域归属校验

**转移条件**：

- 所有声明的域归属校验通过 → 进入活跃状态
- 如果校验失败，实体仍会存在，但不会参与相关域的计算

### 活跃（Active）

实体正常活动，参与仿真。

**特点**：

- 实体参与域的计算
- 可以发出和接收事件
- 状态可以变化

**转移条件**：

- 收到销毁事件 → 进入销毁中状态

### 销毁中（Destroying）

实体正在销毁，处于过渡阶段。

**特点**：

- 实体仍然存在
- 可以执行销毁动画或过渡效果
- 通常由定时器触发进入已销毁状态

**转移条件**：

- 销毁动画完成（由定时器触发）→ 进入已销毁状态

### 已销毁（Destroyed）

实体已销毁，等待清理。

**特点**：

- 实体不再参与计算
- 等待从系统中移除

**转移条件**：

- 清理阶段 → 实体从系统中移除

## 状态转移图

```
[创建] --> 已创建
已创建 --> 活跃（初始化完成）
活跃 --> 销毁中（收到销毁事件）
销毁中 --> 已销毁（销毁动画完成）
已销毁 --> [清理移除]
```

## 生命周期事件

### 销毁事件

触发实体进入销毁流程的事件。

**销毁原因**：

- 被击中
- 超时
- 主动销毁

### 销毁事件的处理

当收到销毁事件时：

1. 将实体状态设为销毁中
2. 设置销毁动画定时器
3. 定时器到期后发出已销毁事件
4. 已销毁事件处理时将实体状态设为已销毁

## 清理流程

在仿真循环的清理阶段：

1. 收集所有已销毁状态的实体
2. 从所有域中移除这些实体
3. 取消这些实体的定时器
4. 从实体存储中移除这些实体

## 生命周期与域的关系

### 域对生命周期的感知

域在计算时通常会过滤掉非活跃状态的实体：

- 已创建：可能参与计算（取决于域的实现）
- 活跃：正常参与计算
- 销毁中：通常不参与计算（取决于域的实现）
- 已销毁：不参与计算

### 实体脱离域

当实体被销毁时：

1. 实体进入销毁中状态
2. 各域的脱离接口被调用
3. 域清理与该实体相关的数据

## 设计考量

### 为什么需要销毁中状态

如果直接从活跃到已销毁：

- 无法执行销毁动画
- 状态变化突兀
- 某些清理工作可能需要时间

通过销毁中状态：

- 可以执行过渡效果
- 给系统时间进行清理
- 状态变化更平滑

### 销毁中状态的持续时间

销毁中状态的持续时间取决于：

- 销毁动画的长度
- 清理工作的复杂度

通常通过定时器控制，在固定时间后转移到已销毁状态。

### 生命周期与事件

生命周期状态的变化通常由事件触发：

- 销毁事件 → 活跃到销毁中
- 已销毁事件 → 销毁中到已销毁

这确保了生命周期变化是可追踪的。

## 常见问题

### 实体可以跳过销毁中状态吗

可以。如果不需要销毁动画或过渡效果，可以直接从活跃到已销毁。

### 已创建状态的实体参与计算吗

取决于域的实现。建议：

- 域在准入校验时将实体加入列表
- 实体在已创建状态时就可以参与计算
- 如果需要延迟，可以由域自己控制

### 实体销毁后标识可以复用吗

不建议。实体标识应该是唯一的，即使实体已销毁，其标识也不应该被复用。这有助于：

- 日志追踪
- 事件关联
- 调试分析
