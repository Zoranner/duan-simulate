# 生命周期

生命周期描述实体从创建到销毁的状态变迁过程。本文档描述生命周期的概念、状态和处理流程。

## 什么是生命周期

生命周期是实体存在期间的状态序列。每个实体都有一个生命周期状态，表示其当前所处的阶段。

**为什么需要生命周期**：

- 区分活跃实体和非活跃实体
- 支持销毁前的过渡阶段
- 便于资源管理和清理

## 生命周期状态

### 初始化中（Initializing）

实体刚创建，正在完成域归属校验。

**特点**：

- 实体已分配标识
- 组件已初始化
- 正在执行域归属校验，尚未加入任何域
- **不参与任何域的计算**

**转移条件**：

- 域归属校验完成（无论各域是否全部通过）→ 进入活跃状态

**关于准入失败**：

准入校验失败不阻止实体进入活跃状态。实体会进入 Active 状态，只是不在拒绝它的域的实体列表中。框架会记录哪些域拒绝了该实体（及拒绝原因），供用户排查配置问题。

### 活跃（Active）

实体正常活动，参与仿真。

**特点**：

- 实体参与域的计算
- 可以发出和接收事件
- 状态可以变化

**转移条件**：

- 收到销毁事件 → 进入销毁中状态

### 销毁中（Destroying）

实体正在销毁，处于过渡阶段。

**特点**：

- 实体仍然存在于仿真中
- **不参与域的计算**
- 允许系统执行销毁前的清理工作（如资源回收、关联通知等）
- 通常通过定时器控制过渡期时长

**转移条件**：

- 过渡期结束（由定时器触发）→ 进入已销毁状态

**说明**：过渡期的具体用途由应用层决定，例如播放销毁效果、向关联实体发送通知等。框架只保证在此期间实体不参与计算，其他行为由用户定义。

### 已销毁（Destroyed）

实体已销毁，等待清理。

**特点**：

- 实体不再参与计算
- 等待从系统中移除

**转移条件**：

- 清理阶段 → 实体从系统中移除

## 状态转移图

```
[创建] --> 初始化中
初始化中 --> 活跃（域归属校验完成）
活跃 --> 销毁中（收到销毁事件）
销毁中 --> 已销毁（过渡期结束）
已销毁 --> [清理移除]
```

## 生命周期事件

### 销毁事件

触发实体进入销毁流程的事件。

**销毁原因**：

- 被击中
- 超时
- 主动销毁

### 销毁事件的处理

当收到销毁事件时：

1. 将实体状态设为销毁中
2. 设置过渡期定时器
3. 定时器到期后发出已销毁事件
4. 已销毁事件处理时将实体状态设为已销毁

## 清理流程

在仿真循环的清理阶段：

1. 收集所有已销毁状态的实体
2. 从所有域中移除这些实体
3. 取消这些实体的定时器
4. 从实体存储中移除这些实体

## 生命周期与域的关系

### 域对生命周期的感知

框架保证域只处理活跃状态的实体：

- 初始化中：**不参与计算**（尚未加入域的实体列表）
- 活跃：正常参与计算
- 销毁中：**不参与计算**（框架在进入此状态时从域的实体列表中移除）
- 已销毁：不参与计算

### 实体脱离域

当实体被销毁时：

1. 实体进入销毁中状态
2. 各域的脱离接口被调用
3. 域清理与该实体相关的数据

## 设计考量

### 为什么需要销毁中状态

如果直接从活跃到已销毁：

- 没有过渡期，无法执行销毁前的必要处理
- 关联系统来不及响应实体的消失
- 清理工作可能需要跨帧完成

通过销毁中状态：

- 提供明确的过渡期，应用层可以在此期间做任何必要的处理
- 给系统时间进行清理
- 状态语义清晰，便于调试

### 销毁中状态的持续时间

销毁中状态的持续时间取决于应用层的需求，通常通过定时器控制，在固定时间后转移到已销毁状态。

如果不需要过渡期，也可以由销毁事件处理器直接将实体设为已销毁状态，跳过这个阶段。

### 生命周期与事件

生命周期状态的变化通常由事件触发：

- 销毁事件 → 活跃到销毁中
- 已销毁事件 → 销毁中到已销毁

这确保了生命周期变化是可追踪的。

## 常见问题

### 实体可以跳过销毁中状态吗

可以。如果不需要过渡期，可以直接从活跃到已销毁。

### 初始化中状态的实体参与计算吗

不参与。初始化中状态的实体尚未加入任何域的实体列表，域在计算时不会处理这些实体。域归属校验完成、实体进入活跃状态后，才开始参与对应域的计算。

### 实体销毁后标识可以复用吗

不建议。实体标识应该是唯一的，即使实体已销毁，其标识也不应该被复用。这有助于：

- 日志追踪
- 事件关联
- 调试分析
