# 事件处理器

事件处理器（EventHandler）负责处理特定类型的事件，是状态更新的执行者。本文档描述事件处理器的概念、注册方式和使用规范。

## 什么是事件处理器

事件处理器是用户定义的函数或对象，与特定的事件类型绑定。当该类型的事件在事件处理阶段被取出时，框架调用对应的处理器。

事件处理器是**唯一合法修改实体状态的地方**。域的计算阶段只读，所有实体状态的变更都通过事件处理器来完成。

## 处理器的职责

事件处理器可以执行：

- **更新实体状态**：修改组件数据
- **创建新实体**：在仿真中生成新的实体
- **销毁实体**：将实体的生命周期状态置为销毁中或已销毁
- **产生新事件**：向事件通道写入新事件（新事件在下一帧处理）
- **操作定时器**：创建、取消定时器

## 注册事件处理器

事件处理器在世界初始化时注册，通过事件类型与处理逻辑绑定：

```
// 注册示例（伪代码）
world.onEvent<MoveEvent>(function(event, world):
    entity = world.entities.get(event.entityId)
    position = entity.get<PositionComponent>()
    position.value = event.newPosition
)

world.onEvent<DestroyEvent>(function(event, world):
    entity = world.entities.get(event.entityId)
    entity.lifecycle = Lifecycle.Destroying
    world.timers.schedule(entity, DESTROY_DELAY, DestroyedEvent { entity.id })
)
```

**注意**：每种事件类型可以注册多个处理器，框架按注册顺序依次调用。

## 处理器的执行上下文

事件处理器在执行时可以访问：

- **实体存储（读写）**：可以读取和修改实体数据
- **事件通道（只写）**：可以写入新事件（下一帧处理）
- **定时器管理器**：可以创建和取消定时器
- **仿真时钟（只读）**：可以读取当前仿真时间

与域的计算接口不同，事件处理器拥有对实体存储的**写入权限**。

## 关于新事件的产生

事件处理器产生的新事件遵循以下规则：

- 新事件写入事件通道后，**在下一帧的事件处理阶段处理**
- 本帧的事件处理阶段不会处理这些新事件
- 这确保单帧内的处理是有界的

如果需要在同帧内的计算阶段就能感知到某个状态变化，应该直接修改实体状态，而不是依赖事件链式触发。

## 定时器的创建

定时器通常在事件处理器中创建，用于延迟触发后续事件：

```
// 在销毁事件处理器中创建定时器（伪代码）
world.onEvent<DestroyEvent>(function(event, world):
    entity = world.entities.get(event.entityId)
    entity.lifecycle = Lifecycle.Destroying

    // 创建定时器，N 秒后触发已销毁事件
    world.timers.schedule(
        entity = entity,
        delay = DESTROY_TRANSITION_SECONDS,
        event = DestroyedEvent { entityId: entity.id }
    )
)
```

定时器关联到实体，实体被清理时定时器自动取消。

## 处理器的设计建议

### 保持简单

处理器应该只做与该事件直接相关的事情。复杂的跨实体计算应该在域中完成，处理器只负责把计算结果写入状态。

### 避免读取其他域的状态

处理器执行时，当前帧的域计算已经结束。如果处理器需要调用域的服务，需要注意：当前读到的是**上一帧**的计算结果。如果需要基于当前帧的计算结果来决定状态更新，应该在域的计算阶段完成判断并包含在事件数据中。

### 事件数据应该自包含

事件携带的数据应该包含处理器所需的全部信息，避免处理器在执行时还需要反查大量其他数据。

## 设计考量

### 为什么状态更新只在处理器中进行

如果域可以直接修改实体状态：

- 不同域的执行顺序会影响计算结果（A 域修改了状态，B 域读到的就是修改后的值）
- 状态变化难以追溯（没有事件记录）
- 回放困难

通过事件处理器统一更新状态：

- 计算阶段所有域看到的是同一帧开始时的状态
- 所有状态变化都有对应的事件记录
- 便于调试和回放

### 为什么处理器在框架初始化时注册

处理器在初始化时注册，而不是动态添加，原因：

- 保证处理顺序的确定性
- 避免运行时注册带来的竞争条件
- 处理器映射表稳定，便于优化查找
