# 事件

事件（Event）是域之间通信的机制。本文档描述事件的概念、类型和处理流程。

## 什么是事件

事件是域的计算结果的表达方式。当域完成计算后，通过发出事件来通知仿真系统。

**核心特征**：

- 事件是单向的：域发出事件，不等待响应
- 事件是异步的：发出后不会立即处理
- 事件是可追踪的：所有事件都可以被记录和分析

## 事件的作用

### 解耦

域不需要知道谁会处理它的事件，也不需要知道其他域的状态。域只需要发出事件，由仿真系统统一处理。

### 可追溯性

所有事件都可以被记录，用于：

- 调试：追踪状态变化的原因
- 回放：重现仿真过程
- 分析：统计和挖掘仿真数据

### 状态一致性

通过将计算和状态更新分离，确保：

- 计算阶段所有域看到的状态是一致的
- 状态更新在统一的阶段进行
- 避免因执行顺序导致的不一致

## 事件的类型

事件类型完全由用户定义，框架不预设任何事件类型。

常见的事件类型包括：

### 状态变化事件

描述实体状态的变化。

例如：

- 移动事件：实体位置变化
- 属性变化事件：实体属性值变化

### 关系事件

描述实体之间的关系变化。

例如：

- 探测事件：一个实体探测到另一个实体
- 命中事件：一个实体击中另一个实体

### 生命周期事件

描述实体的生命周期变化。

例如：

- 销毁事件：触发实体进入销毁流程
- 已销毁事件：过渡期结束，实体进入已销毁状态

## 事件的内容

每个事件包含：

### 事件类型

标识这是什么类型的事件，决定如何处理。

### 相关实体

事件涉及的实体标识。

### 时间戳

事件发生时的仿真时间。

### 事件数据

事件的具体内容，不同类型的事件有不同的数据结构。

## 事件通道

事件通道是事件的临时容器。计算阶段域将事件写入通道，事件处理阶段事件被逐一取出并处理，每帧结束后通道清空。详见 [仿真循环](../architecture/simulation-loop.md)。

## 事件处理器

事件处理器是与特定事件类型绑定的处理逻辑。域在计算阶段有权直接修改自身管辖实体的组件状态（这是"域是权威"的直接体现）；事件处理器负责处理**跨边界**的状态变更——包括生命周期操作（实体创建/销毁）、对非本域实体的影响，以及来自外部观察者的反馈。

### 处理器的职责

事件处理器可以执行：

- **更新实体状态**：修改组件数据
- **创建新实体**：在仿真中生成新实体
- **销毁实体**：将实体生命周期状态置为销毁中或已销毁
- **产生新事件**：向事件通道写入新事件（新事件在下一帧处理）
- **操作定时器**：创建、取消定时器

### 注册与执行上下文

事件处理器通过 `step_with` 的闭包参数传入，无需提前注册。闭包签名为 `|event: &dyn CustomEvent, world: &mut World|`，每次调用时传入当前事件和完整的可变世界引用。

闭包执行时，框架已将当帧所有事件从通道中取出，因此闭包持有的 `world` 引用无其他活跃借用，可以安全调用以下操作：

- `world.spawn(entity)` — 创建新实体并加入仿真
- `world.destroy(id, duration)` — 将实体进入销毁流程
- `world.get_entity(id)` / `world.get_entity_mut(id)` — 读写实体状态
- `world.time()` — 获取当前世界时间

以下操作不应在闭包中调用：

- `world.register_domain(...)` — 域注册只在初始化阶段进行，仿真循环开始后调用行为未定义

### 处理器产生的新事件

事件处理器可以产生新事件，但新事件**在下一帧的事件处理阶段才被处理**，不在本帧处理。这保证单帧内的处理是有界的，不会因事件链式触发导致无限循环。

### 处理顺序

事件按它们被添加到事件通道的顺序处理，域的执行顺序决定了事件的先后。

### 实体生效时序

在事件处理器中通过 `world.spawn()` 创建的实体，会立即完成域归属（`try_attach` / `on_attach` 在 `spawn` 时执行），生命周期状态立即为 `Active`。但由于事件处理阶段在本帧所有域计算**之后**，当帧的域 `compute` 已经执行完毕，新实体**在下一帧的计算阶段才会首次被域处理**。

这是有意为之的设计：同一帧内所有域的计算基于一致的实体快照，不受本帧晚期 spawn 的影响。设计含义：

- 若处理器在帧尾 spawn 了一个导弹，该导弹本帧静止，下帧才开始被运动域积分
- 事件携带的数据应包含新实体所需的完整初始状态（位置、速度等），而非依赖处理器后续查询

## 事件与服务调用的选择

框架提供两种域间通信机制——事件和服务调用，两者互补，各有适用场景：

| | 事件 | 服务调用 |
|---|---|---|
| 时序 | 跨帧（异步） | 同帧（同步） |
| 典型用途 | 状态变化通知、触发实体 spawn/destroy | 查询当前帧计算结果驱动本域决策 |
| 耦合度 | 松耦合（发出方不关心谁处理） | 紧耦合（调用方依赖具体域类型） |
| 典型示例 | FireEvent、HitEvent、ShipDestroyedEvent | `distance()`、`is_hostile()`、`get_detected()` |

**选择准则**：

- **用事件**：需要通知"发生了某件事"，且响应可能涉及世界状态变更（spawn/destroy/伤害扣减），或需要支持日志记录、回放、外部观察
- **用服务调用**：需要查询另一个域"当前帧计算出的状态"来驱动本域的决策，结果需要同帧可用

两种机制经常并存于同一场景。以探测→战斗链为例：探测域发出 `DetectionEvent`（通知事件，供日志/回放消费）的同时，战斗域在 `compute` 中调用 `DetectionRules::get_detected(id)` 服务（同帧查询）做出开火决策——两件事的职责不同，都是正确的设计，`DetectionEvent` 并非多余。
